导读。最强的视频-语言模型（VLM）今天仍是闭源的；而所谓“开源”的强模型，要么偷偷用闭源 VLM 合成数据（本质是在蒸馏），要么干脆不公开训练配方与数据。于是社区缺的不是“又一个权重”，而是一整套能让人接着往上做的地基。Molmo2 的赌注是：把地基彻底摊开——9 个全新数据集、训练代码、三段式配方、权重全部开放，且所有数据都不经过闭源模型蒸馏。更关键的是它要补上当前 VLM 共同缺的一项能力——grounding（指点与追踪）：不止回答“视频里有什么”，还要回答“在第几秒、画面哪个位置”。立题问题因此很具体：当你把一段视频拆成几百帧、每帧再拆成几百个图像 token，序列长度立刻爆炸、不同任务的输出长度又差几个数量级——你要怎么把这堆异质样本喂进一个 LLM，既不浪费算力、又不让“长输出任务”把损失吃干净，还能让模型学会在时空里‘点一下’？本文给出的三块支点是：packing + message-tree（把短样本拼成长序列、把同一视频的多条标注编成共享前缀的树，靠一张定制注意力掩码隔离）、token 加权（给超长输出按数量级压权重）、以及把图像里成熟的 2D pointing 范式延拓到时间轴。最好的 8B 模型在短视频、计数、字幕上是开源开数据阵营第一，视频指点 38.4 F1、追踪 56.2 J&F 甚至压过 Gemini 3 Pro。

一句话贡献Molmo2：首个权重、数据、代码全开且不蒸馏闭源模型的 SoTA 视频-语言模型族；核心是 9 个新数据集（视频密集字幕、长视频 QA、开放词表视频指点/追踪、多图）+ 一套 packing/message-tree/token 加权的高效训练配方。
关键机制视频 = 按 2 fps 抽帧、每帧当“单图 crop”编码 → 三段式训练（图像预训练 → 图文混合 SFT → 短长上下文 SFT）；用 packing 把多样本拼进一条 16k 序列、用 message-tree 把同一视频的多条标注编成共享视觉前缀的树，定制掩码隔离分支，SFT 阶段一条序列塞进 3.8 个样本、训练吞吐 ×15。

一、核心问题：为什么“开源视频 VLM”这件事，绕不开“数据从哪来”和“grounding 怎么进视频”

问题。视频已经是手机、家用摄像头、自动驾驶里最普遍的数据形态，可最强的视频-语言模型（GPT-5、Gemini、Claude）全是闭源的——权重、数据、配方都不公开。那为什么开源社区一直追不上？

引导。表面看是“算力差距”，但论文指出更深的两道坎。第一道是数据的“原罪”：现有大规模视频指令数据，绝大多数是“先把视频切片、用一个强 VLM 给每片配字幕、再让 LLM 合成 QA”——而那个“强 VLM”往往是 GPT-4 这类闭源模型。于是所谓开源数据是闭源模型的蒸馏产物，把闭源模型的偏置原封不动灌进了开源训练集，社区既无法审计、也无法真正超越源头。第二道坎是能力缺口：当前 VLM（连闭源的）都缺 grounding。“机器人抓了几次红色方块？”需要在空间+时间上为每次抓取吐出一个点；“杯子什么时候掉下桌的？”需要返回一条轨迹。图像 grounding 已是标配，视频 grounding 只有少数闭源系统支持，且形式很受限。

回答。所以核心问题不是“再训一个更大的视频 VLM”，而是：能不能在完全不蒸馏闭源模型的前提下，自建一套足够强、且带时空 grounding 的视频数据与训练配方，让社区拿到一个可审计、可接续的地基？这把问题从“刷榜”转成了“造地基”，也解释了为什么本文的主体是数据工程 + 训练系统，而不是某个新网络结构——架构上 Molmo2 反而刻意走“最标准”的 ViT+connector+LLM。

Takeaway。“开源”有三个层级：开权重、开数据、开配方。多数“开源 VLM”只开权重；少数开数据的又被闭源蒸馏污染。Molmo2 要的是三层全开 + 数据不蒸馏，并顺手把 grounding 这块连闭源都没做好的硬骨头啃下来。

二、贡献拆解：作者明确提出的，与读者能顺势推断的

作者明确提出的贡献

  1. 9 个全新数据集（全程不蒸馏闭源模型）。含：2 个开放词表视频指点/追踪集（520k 实例）；1 个密集视频字幕集（104k 视频，平均 924 词/视频，是已知最密的）；2 个长格式 QA（212k）；2 个长视频 QA（约 1.3M）；2 个多图集。合成那一支用的也是自家字幕模型，不碰 GPT-4 等闭源。
  2. 一套高效训练配方。on-the-fly packing（把短样本拼成长序列免 padding）+ message-tree（同一视频多标注编成共享视觉前缀的树）+ 定制块状注意力掩码，SFT 阶段一条 16k 序列平均塞进 3.8 个样本，训练效率 ×15。
  3. 把 2D pointing 延拓到时空。模型能在某一帧点出物体，也能标出“某动作发生的那一刻+位置”，还能连续追踪一个物体——并用一种纯文本压缩坐标格式（归一化 x,y + 时间戳/图像索引 + 物体 ID）统一表达指点、计数、追踪。
  4. 两个被验证有效的训练技巧：视觉 token 间的双向注意力，以及一种token 加权策略（长字幕 0.1、指点 0.2、其余 $4/\sqrt{n}$），平衡“单 token 选择题”与“四千 token 长字幕”之间的损失失衡。
  5. 三个尺寸全开：4B / 8B 基于 Qwen3，7B（Molmo2-O）基于 OLMo——后者用来证明“连语言模型底座也可以是全开的”。

读者能顺势推断的

  • “架构最标准、功夫全在数据与系统”是一种刻意的可复现性设计。把网络结构压成 ViT+connector+LLM 的最朴素形态，意味着别人换个底座也能复刻——novelty 被刻意放在“数据 + 训练系统”这两个最容易开源、最容易接续的层面。
  • pointing 不只是个新任务，更像是计数与追踪的“公共底层表示”。后面消融会看到：先指点再数数，远好于直接预测一个整数；追踪也从指点数据里获得正迁移。pointing 像是 grounding 能力的“原子操作”。
  • “不蒸馏”是约束也是卖点。它逼着作者重做数据管线（人工口述字幕、人机协作 QA、自家模型合成），代价是工程量巨大，回报是数据可审计、且不继承闭源偏置。

三、数据：9 个新数据集怎么在“不蒸馏”的约束下造出来

问题。不让用闭源 VLM，密集视频字幕、长视频 QA、视频指点这些“量大又要细”的数据要从哪来？

引导。作者按“人工标注 / 合成 / 改造已有公开数据”三类，造出 5+4+2 共 11 个数据集（其中 9 个是全新的）。下面按角色拆。

3.1 人工标注（5 个）

  • Molmo2-Cap（密集字幕）。104k 视频级 + 431k 片段级字幕。关键技巧来自 PixMo-Cap：让标注者口述而非打字（口述能塞进更多细节），用 Whisper-1 转写、再由纯文本 LLM 润色成连贯文本；最后用 Molmo（自家图像模型）补帧级细节，LLM 合并片段与帧字幕成一条长字幕。平均 924 词/视频，对比 ShareGPT4-Video 的 280、LLaVA-Video-178K 的 547——这是“为什么 Molmo2 字幕强”的数据根因。
  • Molmo2-AskModelAnything（人写 QA）。140k 人工 QA。先按字幕把视频聚成 31 类、均匀采样保多样性；标注者写细粒度问题（文字/动作/时序关系），刻意避开计数题（计数交给指点数据处理）。每题先让 LLM（Claude Sonnet 4.5）基于自家早期字幕给初答，标注者再接受或迭代修订。注意：这里 LLM 只是辅助标注的草稿工具，不是数据来源——最终答案由人定，且会过滤掉非英文/不匹配/计数题。
  • Molmo2-VideoPoint（视频指点）。280k 视频上 650k+ 指点查询，平均 6 点/视频，覆盖 8 类（物体、动物、动作/事件、指代表达、间接指代、空间指代、比较指代、生成视频的视觉瑕疵）。查询由 LLM 基于自家字幕生成，标注者先定位物体出现的帧、再点其精确位置（2 fps 抽帧）。
  • Molmo2-VideoTrack（点式追踪）。3.6k 片段、15k 复杂自然语言查询，平均 2.28 物体/查询。沿 Ref-VOS 思路：展示已有的分割/框追踪轨迹，让标注者写“只指向其中一部分物体”的非平凡查询，再单独验证一轮。
  • Molmo2-MultiImageQA（多图 QA）。把字幕高相似的图聚成 2–5 张一组（均 2.73 张），人写跨图问题，答案走同一套人机协作环。共 45k 图组、72k QA。

3.2 合成（4 个，关键：合成器也是自家模型）

  • Molmo2-CapQA / -SubtitleQA。用在 Molmo2-Cap 上训出的自家字幕模型给 YouTube 视频配字幕（按场景分段配，逼出细节），LLM 据字幕+元数据生成 1M QA；SubtitleQA 再加 Whisper-1 转写的台词，生成 300k 需“看画面+听台词”的 QA。这一步是“不蒸馏”的核心招：把通常依赖闭源 VLM 的“片段字幕”那一环，换成自家全开模型。
  • Molmo2-MultiImagePoint / -SynMultiImageQA。对 PixMo-Points 软聚类成多图组，用 LLM 把各图标签归一成一个“规范描述”作为跨图指点目标；训练时随机回采原始人工标注而非总用规范标签，保留词汇多样性。SynMultiImageQA 则借 CoSyn 造 188k 富文本（图表/表格/文档）多图样本。

3.3 改造已有公开数据（2 个）

AcademicVideoPoint / AcademicVideoTrack：把 6 个已有追踪标注转成 49k 指点/计数 QA（取物体首现帧时间戳 + 在掩码内按高斯采点）；追踪侧复用 7 个 Ref-VOS 数据集，并用 SAM-2 把 11 个框追踪集补出分割掩码与点任务。

读表要点：采样率不等于数据量占比——视频榜“收敛快”被下采样、视频指点“收敛慢”被上采样。换句话说，采样率是按“每类还能挤出多少增益”手工调的，不是按数据多少摊的。这正是后面 token 加权要解决的同一类“失衡”问题在数据层的对应物。

四、Pipeline / 架构：视频怎么变成 LLM 能吃的 token，输出怎么变回时空里的点

问题。视频比图像多了“时间”这一维，token 数会随帧数线性涨。Molmo2 怎么在不爆算力的前提下把视频塞进 LLM？

引导。核心选择是“不另设视频编码器，把每帧当单图 crop 编码”。论文在相关工作里给了理由：用视频编码器多帧联编，效果反而落后于用 SigLIP 2 这类图像编码器逐帧编。代价是 token 随帧数线性膨胀——于是省算力的活全压到后面的 packing 与池化上。

4.1 逐阶段拆解

  • Cropping（控 token 数的第一道闸）。图像用 1 个降采样全图 crop + 至多 $K$ 个重叠 crop 拼高分辨率（训练 $K{=}8$、推理 $K{=}24$）。视频按 $S{=}2$ fps 抽帧、每帧当单 crop（必要时降采样），最多 $F{=}128$ 帧（长上下文阶段 $F{=}384$）；超长则均匀抽 $F$ 帧，且永远保留最后一帧（多数播放器停在末帧，对用户可能有特殊意义）。
  • Connector（控 token 数的第二道闸）。取 ViT 倒数第 3、第 9 层特征；图像把 $2{\times}2$ patch 窗口用一个多头注意力池化成 1 个向量（以 patch 均值作 query），视频帧改用 $3{\times}3$ 窗口以进一步压 token 数；图像/视频共享 connector 参数，最后过共享 MLP 投影。
  • LLM 输入编排。视觉 token 与文本时间戳（视频）或图像索引（多图）交错；多 crop 图像加列 token 标记长宽比；加图像/帧起始 token；有台词就作为文本接在视觉输入之后。关键一招：允许图像 token（哪怕来自不同帧）相互“前向注意”——即视觉 token 间双向注意（见方法）。
  • 输出。统一的压缩坐标格式：归一化 $x,y$ + 时间戳/图像索引 + 每个不同物体一个整数 ID（用于追踪与计数），点按 时间/索引 → $x$ → $y$ 排序。同一格式同时承载指点、计数、追踪。

数据流 vs 模型流要分清：抽帧、cropping、池化属于把视频压成 token 序列的数据流；packing/message-tree（下一节）是把多个样本拼进一条训练序列的训练流；推理时则用 384 帧、贪心解码（长输出场景才用 top_p 采样）。

五、方法核心：packing + message-tree 的 ×15、token 加权、双向注意力——三块支点逐一拆到底

这一节按 kexue 的姿态来写：每块支点先讲为什么需要，再讲机制，最后从≥2 个独立角度交叉验证它确实成立。

5.1 Packing + Message-tree：为什么能 ×15，掩码为什么是对的

问题（动机先行）。样本长度从几百 token（纯文本/小图）到 16k+（带台词的长视频）跨了两个数量级。若按最长样本对齐做 batch，大量算力会浪费在 padding 上；更糟的是，同一段视频常配 4 条标注（多个 QA、多个指点查询），若把每条标注都当独立样本，那段视频的几百个视觉 token 就要重复编码 4 次。怎么办？

引导（机制）。两招叠加。(1) Packing：一个 on-the-fly 算法从内存里一小池样本中，贪心拼出“尽量塞满”的长序列，免 padding；难点在 VLM 要同时打包 ViT 的 crop 和 LLM 的 token，还要兼容不同的“图像→token”转换方式，作者把它做进了标准 PyTorch dataloader。(2) Message-tree：把“一段视频 + 它的多条标注”编成一棵树——视觉输入作为第一条共享消息（树根），每条标注是一个分支。整棵树线性化成一条序列，配一张定制注意力掩码：每个分支都能注意到共享的视觉前缀，但分支之间互相屏蔽。

回答（为什么省）。省算力有两个相互独立的来源，必须分开算：
来源 A：视觉前缀只编码一次。平均每个样本有 4 条标注，message-tree 让这 4 条标注共享同一份视觉 token，省下的是“把同一段视频重复前向 ViT 3 次”的开销。
来源 B：packing 把序列塞满。SFT 阶段一条 16,348 token 的序列平均能装进 3.8 个样本，几乎没有 padding 浪费。
两者叠加，论文报出训练效率 ×15。注意：这是“吞吐/有效样本”的系统级倍数，不是模型质量的倍数——它解释的是“为什么 Molmo2 用学术级而非工业级算力也能训完”。

Takeaway。×15 不是某个 kernel 的魔法，而是“共享前缀（树）+ 塞满序列（packing）”两个正交节省的乘积。前者砍重复编码，后者砍 padding。

交叉验证（第二条独立路径）：掩码的正确性，可以不看实现、只从“等价性”论证。设想一个对照基线：把每条标注都当独立样本、各自带一份完整视觉前缀、分开前向。把这些独立样本的注意力矩阵块对角地拼起来，正好就是图 3 那张掩码——因为独立样本之间本就没有注意力（块外全 0），同一样本的不同标注本就看不到彼此（这正是“各自独立”的定义）。所以 packed+masked 训练与“逐样本独立训练”在前向数值上等价，差别只在共享前缀省了重复计算、且免了 padding。两条路径（自顶向下的“等价于独立训练”+ 自底向上的“掩码每块的语义”）都指向同一结论：掩码是对的，省的是冗余而非信息。

5.2 Token 加权：为什么长输出必须按数量级压权重

问题（动机）。数据里既有“单 token 输出”的选择题，也有“4000+ token”的长视频字幕。语言建模的损失是对每个输出 token 求和（或平均）的——一条长字幕哪怕采样很少，它贡献的 loss token 数也能轻松压过成百上千条选择题。结果：模型在长输出上越练越好，短答/选择题反而退化。

引导（机制）。给样本的损失乘一个权重。长视频字幕固定 0.1、指点固定 0.2（这两类输出又长又密），其余任务按启发式

$$w(n) = \frac{4}{\sqrt{n}}$$
式 1：token 加权启发式，$n$ 为该样本的答案 token 数。

公式读法。$n$ 越大、权重越小，但只按 $\sqrt{n}$ 衰减、不按 $n$ 衰减。直觉上，若一个样本有 $n$ 个答案 token，它对总损失的“原始贡献”正比于 $n$；乘上 $4/\sqrt{n}$ 后，有效贡献正比于 $\sqrt{n}$——长样本仍比短样本贡献多（毕竟信息量确实更大），但被从“线性碾压”拉回到“平方根温和领先”。这是“既不让长样本独霸损失、又不把长样本压成和单 token 一样轻”的折中。

回答（为什么是 $\sqrt{n}$ 而不是 $1/n$）。若用 $1/n$（即对每个样本的 token loss 取平均），所有样本不论长短对损失的贡献都被拉平成相等——这会过度压制长字幕，丢掉“长输出本就该多学一点”的信号。$4/\sqrt{n}$ 留了一个中间地带：贡献 $\propto\sqrt{n}$，长样本仍领先但不碾压。而字幕/指点这两类即便如此仍嫌太重（输出又长又密、且与短答任务冲突最大），所以干脆给固定小权重 0.1/0.2 兜底。

Takeaway。token 加权解决的是“损失被长输出主导”这个失衡，和数据层“按收敛速度调采样率”是同一思想在不同层面的两次出手。

交叉验证（第二条独立路径）：消融数字直接印证“失衡确实存在且加权确实救场”。表 8(b) 里去掉 token 加权（No token weighting），QA 平均从 64.8 掉到 64.0，而字幕 F1 反而从 39.5 升到 40.0——这正是机制预测的方向：不加权时长字幕独霸损失，于是字幕略好、QA 受损。加权后是“QA 救回来、字幕略让一点”，与“防止长输出主导”的动机完全吻合。一条来自“损失求和的数学结构”、一条来自“去掉它就 QA 掉/字幕升”的消融，两路指向同一结论。

5.3 视觉 token 双向注意力：为什么让帧之间互相“看”有用

问题（动机）。LLM 默认是因果（单向）注意力：第 $t$ 个 token 只能看它之前的。可视觉 token 不是“按时间生成的语言”，而是“同时摆在那里的画面”——强行让前面的帧看不到后面的帧，合理吗？

引导（机制）。Molmo2 允许图像 token（哪怕来自不同帧/图像）相互前向注意，即在视觉 token 内部开双向注意力，文本端仍保持因果。直觉：视频理解常需“后面的帧帮忙解释前面的帧”（动作的结果回看动作的起因），双向注意让每个视觉 token 都能聚合整段时间窗的上下文，而不只是过去。

回答 + 交叉验证。（角度一，原理）因果掩码是为“自回归生成文本”设计的，对“一次性给定的视觉证据”并无必要——去掉它等于把视觉 token 之间的全部成对关系都放开，表达力只增不减。（角度二，消融）表 8(b) 里 “No bidir” 让 QA 从 64.8 掉到 64.4、字幕 F1 从 39.5 掉到 38.5，两个指标同时下降——说明双向注意力的增益不是某个任务的偏好，而是“视觉证据该被双向聚合”这一结构性事实。

Takeaway。把“语言模型的因果先验”不假思索地套到视觉 token 上，是一种隐藏的次优；双向注意力只是把这个不该有的约束去掉。

5.4 Point-then-count：为什么“先点再数”胜过“直接吐一个整数”

问题。计数任务最直接的做法是让模型直接输出一个整数。为什么 Molmo2 偏要先把每个目标都点出来、再数点数？

回答 + 交叉验证。（角度一，机制）直接预测整数把“感知（每个目标在哪）”和“计数（共几个）”塞进一个黑箱回归，模型容易靠数据集先验“蒙一个常见数”；先指点则把任务分解成“逐个定位 → 数点”，每个目标都要有空间证据支撑，计数成了指点的副产品。这也解释了为什么 AskModelAnything 在采集时刻意剔除计数题——计数被交给了 pointing 这条更可靠的路径。（角度二，消融）表 9(a)：直接 “Count” 在 Molmo2-VideoCount 上只有 28.1，“Point then count” 达 34.5（BURST-VC 两者都约 61，但 MVC 差距显著）。同时表 9(b) 显示，去掉指点数据后（不在此列），计数与指点能力一并塌掉。两路（“分解任务消除蒙猜”的机制 + 消融数字）一致：pointing 是计数的底座。

把 5.1–5.4 串起来看：Molmo2 的“方法创新”几乎全是系统与表示层面的——怎么打包、怎么算损失、怎么定义坐标、怎么把计数还原成指点。架构主干始终是最朴素的那一条，这正是“为可复现性而设计”的体现。

六、训练账本：三段式配方与它的算力代价

问题。有了数据和系统，怎么把它们排成一条训练流水线？为什么要分三段？

引导。三段式：(1) 图像预训练（密集字幕带长度条件 + 转写预测，用 PixMo-Cap；加 Tulu 的 NLP 数据保语言能力；并加入 PixMo-Points/Count、CoSyn-Point 的指点数据——作者发现预训练阶段就喂指点，能让指点更强更稳）。混合比 60% 字幕 / 30% 图像指点 / 10% NLP，32k 步、batch 128，约相当于在 PixMo-Cap 上跑 4 个 epoch；ViT/connector/LLM 各用独立学习率。(2) 图文混合 SFT：PixMo + Molmo2 全部数据 + Tulu + 公开视频/图像集，按表 1 的类别采样率混合；类内按数据集大小的平方根采样并手工再平衡（下采样大的合成集）。30k 步、batch 128、最大序列长 16,384。(3) 短长上下文 SFT：同一数据混合上把序列长提到 36,864、$F{=}384$、2k 步，用 context parallelism（每样本 8 卡一组、Ulysses attention），并把视频帧编码与注意力池化也分摊到 CP 组里省显存。因长上下文训练开销大，只作为收尾短训。

回答（为什么这样分）。预训练先在图像上把“看图 + 指点 + 说话”的底子打好（指点提前进场是关键决定）；SFT 才把视频、多图一起混进来做联合微调；长上下文单独拎出来做收尾，是因为它最贵、却主要只对“长视频 QA”有用（见消融表 11）。三段的边界，本质是按“哪段算力花得最值”切的。

算力披露。论文给出步数（预训练 32k / SFT 30k / 长上下文 2k）、batch（128）、序列长（16,384→36,864）、context-parallel 组大小（8 卡/样本），但未公布 GPU 型号、总卡数与墙钟时间，故本文不给 GPU-hours 估算（任何此类数字都只能是“按论文披露数字粗略估算，不等同于实际账单”，而这里连可估算的卡数都缺）。这部分按论文原文记“论文未说明”。

九、局限与复现门槛

  • 长视频是公开的短板。长视频 QA 落后最强开权重模型，作者归因于缺开源长（10 分钟+）训练数据 + 算力不足以做大规模超长上下文训练（论文原话）。这是“数据+算力”问题，不是架构问题。
  • OCR 与多模态推理偏弱。DocVQA/InfoQA 略逊于最强开权重模型，MathVista/MMMU 落后——作者直言缺多模态推理训练数据。
  • 逐帧编码的代价。不用视频编码器、每帧当单图编码，token 随帧数线性涨；虽靠 3×3 池化、packing 缓解，长视频的算力天花板仍在，这也是长视频弱的系统性原因之一。
  • 评测口径的注意事项。论文优先用作者公布结果、缺失处自测，但坦承“别人的 eval 细节（prompt、帧数）有时不公开，结果需谨慎解读”。Point-Bench 旧版本曾因评测 bug 给出过偏高分数（论文脚注已更正）。
  • 复现门槛。三段式 + packing/message-tree/context-parallel 是相当重的训练系统工程；序列长 16k→36k、batch 128、每样本 8 卡 CP，属工业级配置。GPU 型号、总卡数、墙钟时间论文未说明，故无法给出可信 GPU-hours。好处是代码与数据全开（github.com/allenai/molmo2），地基本身可审计。

结语：几点学术判断

一句话总结：Molmo2 把“开源视频 VLM 追不上”这道题，从“训个更大的模型”改写成“造一套不蒸馏闭源、且带时空 grounding 的全开地基”——架构刻意走最朴素的 ViT+connector+LLM，把全部创新压在数据工程（9 个新数据集）与训练系统（packing+message-tree ×15、token 加权、双向注意力、point-then-count）上，于是字幕、计数、指点、追踪四项全面领先开源、部分压过闭源，而长视频/OCR/推理的短板也都清清楚楚可溯源到数据与算力。

三个 takeaway：
  1. “省算力”要拆成正交的几块分别省。×15 = message-tree 共享视觉前缀（砍重复编码）× packing 塞满序列（砍 padding），靠一张块对角掩码保证“省的是冗余、不是信息”——它与“逐样本独立训练”前向数值等价。
  2. 损失/采样的“失衡”要在多个层面同时治。数据层按收敛速度调采样率（视频下采、指点上采），损失层按输出长度调 token 权重（$4/\sqrt{n}$，字幕/指点固定 0.1/0.2）——同一思想出手两次，缺一个短答任务就退化。
  3. grounding 是“数据长出来的能力”，pointing 是它的原子操作。计数还原成“先点再数”、追踪从指点正迁移、指点预训练提前进场——给基线临时调 prompt 凑不出来。把 2D pointing 延拓到时间轴这一步，是 Molmo2 把 grounding 带进视频的真正支点。

留给后续的两条验证/反驳路径：(a) 长视频短板究竟是“数据缺”还是“逐帧编码的架构天花板”？把开源长视频数据补到与短视频同量级、并做等算力对照，若差距消失则坐实是数据问题；(b) point-then-count 的优势是否在“极高计数”（几十上百目标）时仍成立，还是会被点序列长度反噬——这关系到 pointing 作为“计数底座”的适用边界。
