tttLRM: can a small network become the memory for long-context 3D reconstruction?

Paper2Html2026-06-25CVPR 2026 digest entry | Adobe Research / UPenn / UCISource: arXiv:2602.20160v2 PDF and supplement
3D ReconstructionLarge Reconstruction ModelTest-Time TrainingLarge Chunk TTTFast WeightsAutoregressive Gaussian SplattingMathJax checked

The short version. tttLRM asks whether a feedforward Large Reconstruction Model (LRM) can keep using more posed images without paying the quadratic cost of full attention. The bottleneck is memory: ordinary attention keeps every key-value pair, while long-view 3D reconstruction needs a compact state that can absorb 16, 32, or 64 views. Test-Time Training (TTT) turns that state into fast weights, a small neural network updated by gradient descent during inference. tttLRM uses Large Chunk Test-Time Training (LaCT) blocks to write posed image tokens into those fast weights, then queries the memory with virtual-view tokens and decodes the result into Gaussian splats or triplanes. On Google Scanned Objects (GSO) at 512 x 512 with 24 input views, it reports 34.80 Peak Signal-to-Noise Ratio (PSNR) in 1.1 seconds versus GS-LRM's 33.26 PSNR in 5.5 seconds, with the same Learned Perceptual Image Patch Similarity (LPIPS) value of 0.022. On scene benchmarks, the single tttLRM model scores above the feedforward Long-LRM rows at 16, 32, and 64 views in PSNR, but at 64 views the per-scene optimizers pull ahead on quality: 3D GS, Mip-Splatting, and Scaffold-GS all beat tttLRM on DL3DV PSNR, and 3D GS has the lowest LPIPS. The autoregressive version updates the same fast weights as views arrive, so the scene can be refined online instead of recomputed from scratch. The boundary is capacity: the fast-weight memory is fixed size, and the supplement shows that harder scenes and longer streams can degrade quality.

One-line contributionA TTT-based LRM that compresses many posed observations into fast weights and decodes that memory into explicit 3D representations.
Key mechanismInput views update the fast weights; virtual views only query them. That read/write split makes long-context feedforward reconstruction and online reconstruction use the same memory interface.
Core evidenceAt 512 x 512 on GSO, tttLRM improves over GS-LRM at 8, 16, and 24 views in PSNR while becoming faster as the view count grows. On DL3DV-140 and Tanks&Temples, one tttLRM model scores above Long-LRM's separate feedforward rows in PSNR.
Reproduction boundaryThe paper discloses 24 LaCT blocks, hidden size 768, curriculum stages, optimizer settings, and 64 NVIDIA A100 80GB GPUs. It does not disclose wall-clock training time.

1. Motivation: why is long-context 3D hard for LRMs?

A Large Reconstruction Model predicts a 3D representation directly from images. That is attractive because it avoids per-scene optimization, which can take minutes. The usual price is context length. If a model fuses views with full attention, then each token interacts with every other token, so the compute grows as $O(N^2 d)$ for $N$ tokens and hidden dimension $d$.

Multi-view 3D reconstruction wants the opposite scaling. More views usually mean better coverage, fewer occlusions, and better geometry. But high-resolution images produce many patches, and each additional camera view adds another block of tokens. A model that handles four views comfortably can become impractical when the scene asks for 32 or 64.

Q: Why not just keep all old tokens and attend over them?

Guide: attention is a precise memory, but it is an expanding memory. Every new view adds more keys and values that future views must compare against.

A: Keeping all tokens preserves information, but it makes every step more expensive. Long-context 3D needs a memory whose cost is bounded. The risk is that a bounded memory becomes too weak and forgets important geometry.

Takeaway: The problem is not only speed. The method needs a compact memory that still preserves enough geometry to render novel views.

tttLRM's answer is to make the memory trainable at inference time. Instead of storing all key-value pairs, it stores a small network that has been updated on those pairs. That gives up the exactness of the attention cache, but it buys linear scaling and a natural online state.

2. Contributions: what the paper actually adds

The authors state three contributions. First, they propose tttLRM, a TTT-based reconstruction model for both long-context feedforward reconstruction and autoregressive, meaning online and causal, 3D reconstruction with linear complexity. Second, they interpret TTT fast weights as a latent 3D memory that can be queried and decoded into explicit outputs such as 3D Gaussian Splatting (3DGS) or triplane-based Neural Radiance Fields (NeRFs). Third, they report the strongest feedforward reconstruction quality in their object-level and scene-level benchmark tables, with the important qualification that optimization-based methods remain stronger in some 64-view scene metrics.

The larger idea is not "TTT replaces attention everywhere." It is narrower and more useful. If the task naturally has repeated observations of the same scene, then a small network updated by gradients can act as a learned compression of the history. The paper's experiments ask whether that compression is strong enough for photorealistic novel view synthesis.

How to read the headline

"Linear complexity" in this page refers to the LaCT update and apply path with respect to sequence length. It does not mean training is cheap: the paper trains large models across curriculum stages on 64 A100 80GB GPUs.

3. Data and metrics: objects, scenes, and novel views

The paper evaluates novel view synthesis, or NVS: given input views with camera poses, the model reconstructs a 3D representation and renders held-out views. For object-level training, it uses Objaverse, with 730K centered objects and 32 rendered views per object. Evaluation uses 100 Google Scanned Objects, abbreviated GSO, with selected input views and the same random 8 views for testing.

For scene-level training, it uses DL3DV-10K, a dataset of 10,510 high-resolution videos with up to 500 keyframes and camera poses from COLMAP. Evaluation uses DL3DV-140, a 140-scene test set, and Tanks&Temples. The paper follows Long-LRM's input and target split: test views are sampled every 8 views, and input views are selected by K-means over camera position and viewing direction.

The metrics are Peak Signal-to-Noise Ratio (PSNR), Structural Similarity Index Measure (SSIM), and Learned Perceptual Image Patch Similarity (LPIPS). Higher PSNR and SSIM are better. Lower LPIPS is better. A row can therefore win PSNR while still lose LPIPS, and this page keeps those comparisons separate.

4. Pipeline: write observations, query virtual views

tttLRM starts with posed red-green-blue (RGB) images and their ray embeddings. A ray embedding tells each pixel where its camera ray starts and points, so image patches carry both appearance and geometry. The model patchifies each image, converts patches into tokens, and sends them through a stack of LaCT blocks.

Q: Why are the virtual views not allowed to update the fast weights?

A: Input views are evidence about the scene, so they should write memory. Virtual views are questions about that memory, so they should only read it. Mixing those roles would let a query camera change the scene state it is trying to inspect.

Takeaway: The key interface is simple: observed views write; virtual views read.

5. Method: TTT as learned compression of a key-value cache

Standard attention projects tokens into queries, keys, and values. Each query compares with all keys, then mixes the values. TTT changes the storage rule. It treats key-value pairs as training data and updates a small function $f_W$ so that it maps keys to values. The updated weights $W$ become the memory.

$$W \leftarrow W-\eta\,\nabla \mathcal{L}_{\mathrm{MSE}}\big(f_W(k),v\big),\qquad o=f_W(q)$$
Eq. 1: TTT fast-weight update and query readout.

How to read Eq. 1

$k$ and $v$ are the key and value vectors from input tokens. $q$ is a query. The loss is mean squared error (MSE), so the update trains $f_W$ to remember the mapping from keys to values. The readout then asks the trained memory what value corresponds to a query. The paper uses this as a compressed replacement for the key-value cache.

LaCT makes this practical at long sequence length by updating over large chunks, up to one million tokens in the cited LaCT work, rather than tiny token minibatches. tttLRM then inserts LaCT into a 3D reconstruction model. The paper writes the tokenization and one block's simplified computation as:

$$T=\mathrm{Tokenize}\!\left(\mathrm{Patchify}\big(\{(I_i,R_i)\}_{i=1}^{N}\big)\right),\qquad T=\{T_{i,j}\mid i=1,\ldots,N;\ j=1,\ldots,HW/p^2\}$$
Eq. 2: posed images and ray embeddings become visual tokens.
$$T_i=T_i+\mathrm{WinAttn}(T_i),\qquad W=\mathrm{Update}\big(\{T_i\}_{i=1}^{N}\big),\qquad T_i=\mathrm{Apply}(W,T_i)$$
Eq. 3: within-view window attention, cross-view fast-weight update, and application back to tokens.

For Gaussian reconstruction, virtual views form the query tokens. For triplane NeRF reconstruction, learnable triplane features play that query role. In both cases, the paper's important constraint is the same:

$$T_{v_i}=\mathrm{Apply}(W,T_{v_i})$$
Eq. 4: virtual tokens query the fast weights without updating them.

The decoded Gaussian parameters are mostly direct: RGB color, scale, rotation, and opacity are predicted from the query tokens. Positions are handled through depth. The decoder predicts pixel depth, converts it to real depth through a range function, and then uses known ray origins and directions to place each Gaussian.

Q: What makes this more expressive than a hand-written linear state update?

Guide: a linear state update stores history by a fixed rule. TTT stores history by optimizing a function on the current chunk.

A: The memory is not just an accumulator. It is a small model trained on the scene's current key-value pairs. That gives the update more flexibility, but it also makes the memory lossy and capacity-limited.

Takeaway: tttLRM trades an exact expanding cache for a learned fixed-size memory.

The loss is still a rendering loss at the outside. The model renders the reconstructed Gaussian scene into target views and compares those images with ground truth:

$$\mathcal{L}_{\mathrm{RGB}}=\mathrm{MSE}(I_{\mathrm{pred}},I_{\mathrm{gt}})+\lambda\,\mathrm{Perceptual}(I_{\mathrm{pred}},I_{\mathrm{gt}})$$
Eq. 5: RGB reconstruction loss with VGG-19 perceptual features.

For scene-level data, the paper adds depth and opacity regularization:

$$\mathcal{L}=\mathcal{L}_{\mathrm{RGB}}+\lambda_{\mathrm{depth}}\mathcal{L}_{\mathrm{depth}}+\lambda_{\mathrm{opacity}}\mathcal{L}_{\mathrm{opacity}}$$
Eq. 6: final scene-level objective.

The load-bearing assumption

The fast weights must retain enough scene geometry after compression. If the memory is too small for the scene, or if online updates drift too far from earlier observations, the method can forget. The supplement's complexity analysis and selective-update ablation test exactly this boundary.

6. Training and compute: what is disclosed?

The main model has 24 LaCT blocks with hidden dimension 768. Window attention uses 64 dimensions per head with query-key normalization. The feedforward layer is a two-layer multilayer perceptron, or MLP, with an intermediate expansion ratio of 4. The architecture shares parameterization with TTT-LVSM except for the decoding module, which lets the authors initialize from pretrained novel-view-synthesis weights.

8. Ablations: what keeps the memory from failing?

8.1 Pretraining transfers from NVS to explicit 3D

The paper initializes from TTT-LVSM, a large novel-view-synthesis model. Table 3 shows that this is not just a convergence trick. It improves the final result for both Gaussian splats and triplanes, with a much larger gain for triplanes.

The GS improvement is modest, 0.37 dB PSNR, while the triplane improvement is 1.47 dB. That difference matters: the transfer claim is strongest for the harder representation shift, not equally strong across both rows.

8.2 Full reconstruction beats predict-and-merge

The autoregressive version could save compute by keeping old Gaussians and only merging newly predicted Gaussians. The paper calls this "Predict & Merge." It is cheaper, but Table 4 shows that it accumulates errors.

Paper Table 4, transcribed. Progressive Gaussian merging versus full reconstruction, compared on 32 views under 1K iterations of fine-tuning.

The point estimate is a 2.13 dB PSNR gap. Mechanistically, this supports the paper's warning about online memory: if old Gaussian errors are frozen into the scene, later views cannot fix them. Full reconstruction costs more, but it lets the current fast-weight state revise the whole representation.

Table 5 isolates a lower-resolution 32-view setting. Muon gives a small PSNR improvement. The depth-plus-opacity row gives another small metric gain and a large reduction in nearly opaque Gaussians.

The supplement tests a training-free selective update that uses historical fast weights and a Fisher-information-style importance estimate to reduce drift. The gain is small but directionally consistent with the memory-capacity story.

Q: Do the ablations agree with the mechanism?

A: Yes, but unevenly. Pretraining gives the memory a useful 3D prior. Full reconstruction avoids freezing accumulated Gaussian errors. Muon and regularization mostly stabilize the update and representation.

Takeaway: The memory works because it is trained, initialized, and regularized as a memory, not because fast weights alone are magic.

9. Limitations: where the result should stop

The paper's own limitation is the right starting point: fast-weight memory has fixed capacity. That is the price of replacing an expanding attention cache. The supplement makes the capacity issue measurable, reporting lower PSNR for outdoor scenes than indoor scenes, 24.45 versus 24.96, and lower PSNR for high-frequency scenes than low-frequency scenes, 24.20 versus 25.97.

Q: What is the easiest overstatement?

A: "TTT solves long-context 3D reconstruction." The paper supports a narrower claim: LaCT fast weights give a strong, seconds-scale feedforward memory for posed multi-view 3D, especially against feedforward baselines.

Takeaway: The method is strongest when the user needs more context than attention can afford and less per-scene optimization than classical 3DGS requires.

10. Summary: the useful mental model

In one sentence: tttLRM replaces the growing attention cache with a learned fast-weight memory, then treats 3D reconstruction as writing posed evidence into that memory and reading it from virtual views.

Three takeaways

  1. Memory is the main object. The novelty is not a new Gaussian decoder alone; it is the use of TTT fast weights as the scene state.
  2. The strongest evidence is matched by view count. Against GS-LRM on GSO and Long-LRM on scenes, tttLRM's long-context PSNR and speed story is strong. Against minute-scale optimization at 64 views, the story becomes speed and competitiveness, not universal quality dominance.
  3. The failure mode is predictable. A fixed fast-weight memory can forget or compress away complex content, so selective updates, pretraining, and full reconstruction matter.

A good reading path through the paper is: read Figure 2 and Eq. 1-4 to understand the memory interface, then Table 1 and Table 2 to separate object and scene claims, then Tables 3-6 to see why pretraining and update design are not optional details.
